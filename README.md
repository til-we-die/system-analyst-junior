### Задание 3: Архитектура PUSH-уведомлений в мобильном приложении «Петрушка Зеленая»

Система для отправки пушей состоит из мобильного клиента, нескольких внутренних микросервисов, брокера сообщений и внешних сервисов доставки от Google и Apple.
#### Основные компоненты системы

**Мобильное приложение** выступает первичным источником данных об устройствах. При первом запуске и каждом новом логине пользователя приложение запрашивает разрешение на отправку уведомлений и получает от операционной системы уникальный токен устройства. Этот токен вместе с идентификатором пользователя (UserID) отправляется на бэкенд. Также приложение отвечает за обработку кликов по уведомлениям: извлекает скрытые данные и открывает нужный экран.

**Бэкенд** представлен несколькими микросервисами. Первый из них - **Device Service**, который принимает и хранит токены в базе данных. Он связывает каждый токен с конкретным UserID и платформой (Android/iOS). Другие микросервисы, такие как **Order Service** (заказы) и **Cart Service** (корзина), занимаются исключительно своей логикой. При наступлении важного события (например, "статус заказа изменился на «Отменен»") они публикуют событие в центральный **брокер сообщений** (Kafka или RabbitMQ). 

**Push Notification Service** (сервис пуш-уведомлений) подписан на нужные топики брокера. Когда он получает событие, то выполняет три ключевые задачи. Во-первых, на основе типа события и данных в нём (например, ID заказа, имя товара) формирует понятный пользователю текст уведомления. Во-вторых, по UserID из события обращается к базе данных Device Service (или к своему локальному кэшу) и получает список активных токенов этого пользователя. В-третьих, отправляет эти токены вместе с текстом и дополнительными скрытыми данными во внешние сервисы **Firebase Cloud Messaging (FCM)** для Android и **Apple Push Notification Service (APNS)** для iOS. Эти внешние сервисы уже отвечают за финальную доставку на устройства.

#### Процесс отправки уведомления

Допустим, пользователь, находясь в приложении, отменяет заказ. Этот сценарий выглядит следующим образом. На первом этапе Order Service фиксирует отмену и генерирует событие `OrderCancelled`, содержащее UserID пользователя и ID отмененного заказа. Событие попадает в брокер сообщений в топик `order_events`. Push Notification Service, подписанный на этот топик, считывает событие. Он формирует заголовок (например, "Заказ отменен") и текст ("Ваш заказ №1 отменен. Средства вернутся в течение 14 дней"). Затем сервис находит все активные токены пользователя (у одного человека может быть несколько устройств - телефон и планшет). В полезную нагрузку для внешних сервисов, помимо текста, помещаются скрытые данные для Deep Link: `{"action": "open_order", "order_id": 1}`. Запрос уходит в FCM и APNS. Если устройство пользователя онлайн, уведомление приходит мгновенно. Если оно выключено, FCM/APNS сохраняют уведомление и доставляют, когда устройство вновь появится в сети. Наконец, когда пользователь кликает по пушу, операционная система передает скрытые данные в приложение, и оно открывает экран деталей заказа с ID 1.
